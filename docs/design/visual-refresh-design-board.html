Grounding

What the current UI actually does

Every BEFORE on this board was verified in source, not remembered. The audit also corrected one item on the original shortlist.

  • Categories are a seed list of 3 income + 29 expense names in entriesHelper.js, plus any the user creates — so an icon map needs a fallback, not just a lookup.

  • Month stepper buttons are 2.5rem circles; at the first/last month with data the button is swapped for an invisible placeholder (visibility: hidden) — the layout holds but the control vanishes.

Candidates

Eight before→afters, each with a verdict

The shortlist's five items survive in some form — one rescoped — the audit surfaced a sixth (tabular figures), and two more were added at the stakeholder's request: the navigation bar (7) and the summary total tile (8). Every AFTER stays inside the existing vocabulary: same tokens, same radii, gold spent sparingly.

Why

The shortlist billed this as "centered title + squared buttons vs left-aligned title + circles." The audit found the title is already centered — so the candidate shrinks to two honest deltas: rounded-square steppers, and keeping the stepper visible-but-disabled at the data edge instead of vanishing it.

Why

Circles appear nowhere else among controls — reserving them for the money chips makes both shapes mean something. A persistent disabled stepper also stops the control from teleporting under your thumb at the data edge.

Trade-off

This is the most-touched control in the app, changed for consistency rather than a user complaint. The disabled-vs-hidden switch is a real behavior change: a disabled button is announced by screen readers where the hidden one was silent.

Validate

Confirm the existing goToPrevMonth/goToNextMonth integration helpers still pass (they query by accessible name, which doesn't change). VoiceOver/NVDA pass on the disabled state; check no accidental taps registered at the edge.

4

Two-line entry rows — only when a description exists

Today the row concatenates Category - description into one ellipsized line. Proposed rule: an entry with a description gets category as a semibold title and the description as muted subtext; an entry without one keeps today's single line. The rule is content-driven, so rows never change shape on resize.

Why

Category becomes the scannable anchor (it pairs directly with candidate 1's glyphs) and long notes get a full line instead of the tail end of one. The hierarchy matches what users actually scan for first: what kind of spend, then which one.

Trade-off

Described rows are ~18% taller, so note-heavy months show fewer entries per screen. Mixed one- and two-line rows also give lists a ragged rhythm — deliberate here, but it must be seen with real data, not this curated sample.

Validate

Load a real backup CSV and count entries-above-the-fold on a 360×780 viewport before/after. Integration tests keep asserting on visible text per the house rule — the category and description remain separately findable by screen.findByText.

5

Whisper-faint gold page glow

One fixed radial wash behind the top of the shell column, at most 5% gold, gone by two-thirds down the viewport. It warms the near-black page without touching the contrast of anything painted on a surface.

Pursue, guarded
Before · proposed · and the failure mode to guard against
July 2026Before — flat #0e1116
July 2026After — 5% gold, top only
July 2026Rejected — 16%: backlight bleed

token: $glow-page: radial-gradient(90rem 40rem at 50% -6rem, rgba(240,185,11,0.05), transparent 62%) · painted once on body, fixed size — never stretches with page height, never repaints on scroll

Why

The page ground is currently the one surface with no brand presence. A top-anchored wash adds depth and quietly locates the accent, at zero cost to content: everything readable sits on surface/raised layers above it.

Trade-off

On OLED at low brightness, faint gradients band or read as screen smudge. And the effect is invisible on some panels — meaning the payoff may be nothing, which is an acceptable outcome only because the cost is one token.

Validate

Check on one OLED phone at minimum brightness and one desktop IPS at full: no visible banding, no "dirty screen" impression. If it fails, drop the candidate — do not fix it by raising the alpha.

6

Tabular figures for money columns

Found during the audit, not on the shortlist. Amounts render in proportional figures, so right-aligned digits wobble from row to row. One declaration — font-variant-numeric: tabular-nums — puts every digit on a fixed width, and the column snaps into a grid.

  • shapeIndicator behind the icon, not the item. A 2.4×1.55rem lozenge at $radius-control — the same radius as buttons and inputs. Circles stay reserved for entry chips; a fully-round 999px pill (as in the reference example) would introduce a radius the token set doesn't have.
  • colorReuses existing state tokens. $accent-soft fill + rgba(240,185,11,.28) hairline + gold icon/label — identical to how the app already signals "active". The searchable category combobox that just landed on master marks its selected row with the same $accent-soft — so "selected = $accent-soft" is now an established convention this candidate follows, not one it invents.
  • motionThe indicator settles on selection — a 140ms fade/scale on $transition-quick, matching every other state change in the app. Optional later enhancement: slide it between tabs. Both gate on prefers-reduced-motion.
  • iconsGlyph = destination. Converges with the reference on grid for Categories; diverges on Buckets — a bucket with a fill line reads as a spending-limit container (the feature's own metaphor) where the reference's shopping bag implies generic spending. Fixed Entries keeps the recurring repeat loop over an ambiguous branch glyph. Drawn in the same 2px-stroke family as candidate 1.
Why

A concentrated indicator reads as "selected" faster than a full-width tint, especially in peripheral vision at the bottom of the screen; and destination-specific shapes make tabs recognizable by glyph, not just label — the same pre-attentive logic as candidate 1, one level up in the IA.

Trade-off

The nav is muscle memory — swapping tag→grid and pie→bucket costs current users a brief relearn. The indicator is a new sub-component (though built entirely from existing tokens). And a bottom bar carries more visual weight once the active tab has a distinct chip.

Validate

Tap target stays the full item (≥44px); confirm the indicator fits inside the 4.25rem bar + safe-area-inset-bottom with no clipping. Verify isActive still lights Home on both / and /dashboard. Integration tests key off accessible names (labels unchanged), so they hold; run the glyphs past candidate 1's glance + colorblind checks.

8

Summary total tile — label/value hierarchy

On /expenses, /incomes and /summary the month total renders today as one flat string — Expenses total: $2,418.90 — label and amount the same size, weight, and color. This gives the single most important number on the screen a proper two-tier layout: a muted label, a large tone-colored value, on a softly tinted card. It reuses the tone treatment the ContentTileSection already carries, and lines up with the total tile in the Filters & Sorting proposal.

  • hierarchyLabel and value stop competing. The label drops to $text-secondary/0.85rem; the amount rises to 1.4rem/700 in the tone color — so the number is read first and the caption second, instead of one undifferentiated colored line.
  • toneSign drives color, per the code that's already there. Summary already computes tile-tone--expense for a negative total, --income for positive, and neutral for exactly zero — so green never misreads as "money in". The refresh keeps that logic and adds a matching soft-tinted background + border.
  • figuresTabular by default here. This is the largest number on the screen and the one people compare month to month — it's the strongest place for candidate 6's tabular-nums, so digits don't reflow as the total changes.
  • scopeThis is the always-present tile, not the filtered banner. The ContentTileSection money tile pre-dates Filters & Sorting; the gold "filtered total" banner in that proposal is a separate, filters-owned element and stays out of scope here.
Why

The month total is the headline figure of these screens, yet today it's typeset exactly like its own caption. A two-tier tile gives the number the weight it already deserves — and the soft tint turns the tone from a subtle text color into something legible at a glance.

Trade-off

Adds a tinted block to the top of three screens — more color up front, justified only because it's the single most important number. Splitting label from value means the two screens pass structured content instead of a joined string, and a few tests that locate the tile by its combined text need updating.

Validate

Contrast-check label ($text-secondary) and value on the soft-tinted grounds; confirm the zero-total neutral case shows no green/rose; verify the /summary tile still flips tone with the sign. Tests keep asserting on visible text — label and value remain separately findable by screen.findByText.

Pressure-tested out

Considered and dropped

Rejections are part of the proposal — these came up during exploration and don't survive contact with the system.

Per-category chip hues. The chip's tint is load-bearing: once arrows become category glyphs (candidate 1), green/rose is the only always-on direction signal. Thirty decorative hues would bury a two-hue semantic system for zero scanning gain.

"Center the month title." On the original shortlist, already true in production — global.scss centers h1.title. Dropped as a claim; the surviving remainder is candidate 2's rescope.

Gradient surfaces / glass blur on cards. Off-vocabulary. The identity is flat graphite layers separated by hairline borders, with gold spent only on action and state. Adding sheen would trade the "calm" out of calm fintech.

Rollout

Systematize what survives — sequenced apart from filters

Four phases, each independently shippable and reviewable, each its own PR with the house version-bump and changelog entry. Order is by risk and by which files they touch — not by the filters timeline.

A

Tokens & primitives no visible change

Extend the system before touching screens: add $glow-page and a money-figures mixin to variables.scss; create the category→icon map module beside the seed lists (with a unit test that every seed category resolves and unknowns hit the fallback); commit the brand SVG source. There's fresh precedent for exactly this: master just promoted the dropdown caret into a shared $select-chevron token so the native selects and the new combobox can't drift — the same single-source-of-truth move these tokens make. Approval of this board gates the phase.
